Microcorruption Guide #0 - New Orleans

µcorruption guide

In this post and the next posts, I’ll try to write companion guide for you when you’re solving the µcorruption CTF.

You can open µcorruption here. You’ll need to create an account to play.

Prerequisite knowledge

I assume you, the reader, are already familiar with these subjects:

  • Basic understanding of computer architecture
  • What a CPU is
  • What are instructions
  • What is RAM
  • What is input-output
  • Basic understanding of debuggers
  • What are registers

If you are unfamiliar with some of these subjects, don’t worry! Playing µcorruption is a good way to learn about them.

After you play the tutorial, we can start with…

New Orleans

Let’s start by going over the main functions: main, create_password, get_password, and check_password.

get_password

44b2 <get_password>
44b2:  3e40 6400      mov	#0x64, r14
44b6:  b012 8445      call	#0x4584 <getsn>
44ba:  3041           ret

First, it looks like all get_password is really doing is getting a string from the user, and nothing interesting is going on here, let’s move on then.

In the next step we want the understand where the magic happens.

create_password

447e <create_password>
447e:  3f40 0024      mov	#0x2400, r15
4482:  ff40 5f00 0000 mov.b	#0x5f, 0x0(r15)
4488:  ff40 3d00 0100 mov.b	#0x3d, 0x1(r15)
448e:  ff40 4200 0200 mov.b	#0x42, 0x2(r15)
...

create_password doesn’t contains any cmp instructions, so we can’t have any impact on this function’s flow. Let’s move on!

Please notice that the password is randomized (probably by the User ID or something). So you will have to find the password from your session - the screenshots here won’t work for you.

check_password

44bc <check_password>
44bc:  0e43           clr	r14
44be:  0d4f           mov	r15, r13
44c0:  0d5e           add	r14, r13
44c2:  ee9d 0024      cmp.b	@r13, 0x2400(r14)
44c6:  0520           jne	#0x44d2 <check_password+0x16>
44c8:  1e53           inc	r14
44ca:  3e92           cmp	#0x8, r14
...

Place a breakpoint at the beginning of the check_password function by running break check_password, then run the debugger with continue. After a couple of instructions we got to the first cmp.b instruction.

Tip - Don’t bother yourself with deep understanding about the sr register state and how the jmp instructions handle it. In short, every cmp and similar instructions are changing the sr register, and every jne instruction and similar instructions decide what to do based on the sr register state. For now, we can assume that our gut feeling about program flow will be just right.

By exploring this instruction’s parameters we can understand that this part of the function compares between our input and the memory that placed in 0x2400. At this point we can just gamble that if we input the string from 0x2400 somthing good will happen. Another insight we can glean from the other cmp instruction that exists in the function is about the compared string length.

Find out what is the string in ASCII format that was generated for your user (remember, it’s at memory address 0x2400). Use it as the lock’s input.

Congrats!
